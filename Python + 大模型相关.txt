1、什么是Python的闭包？
闭包是Python中的独特函数机制，指在一个内部函数中，引用了外部函数的变量，而这个外部函数已经执行完毕并返回了内部函数，然而内部函数仍然可以访问这些外部函数中的变量。
简单来说他也是一种记忆化的机制
def outer_function(x):
    def inner_function(y):
        return x + y
    return inner_function

closure_func = outer_function(10)
print(closure_func(5))  # 输出 15

在这个例子中，我们先执行了外部函数，为x赋值，并返回内部函数。而内部函数的执行依然可以使用包裹他的外部函数。

闭包常用于函数工厂、装饰器和维持状态的场景，允许创建带有状态的可重用函数，而无需依赖全局变量，且基于此，我们的变量管理就可以放在外部函数部分，技能保护外部函数中的内容不会随便修改，也增强了数据的封装性。其使用的范围用在延迟计算或是记忆化技术中，比如我们的缓存，缓存内容可以放在外部函数，这样内部函数的实现如果有外部函数的缓存，就可以加速计算了。

2、Python的装饰器是什么， 用法是什么：
装饰器是一种设计模式，在不需要改变原先本身代码的情况下，进一步修改或是增强函数、方法或是类的功能。
其本质就是一个函数，接受一个函数作为参数并返回一个新的函数，（因为Python中函数可以作为变量传递，且由于闭包的存在，装饰器可以通过嵌套函数保留原函数，并包裹新功能）。

基本装饰器的架构如下： 通常使用@语法应用

比如这样一个要增强wrapper()函数的功能，比如，在函数调用前后自动打印日志，但又不希望修改wrapper()函数的定义，这种在代码运行期间动态增加功能的方式，称之为“装饰器”（Decorator）。

def my_decorator(func):
    def wrapper(*args, **kwargs):
        print("Something is happening before the function is called.")
        func(*args, **kwargs)
        print("Something is happening after the function is called.")
    return wrapper
//这个的意思是把decorator置于函数定义处，作为入参的函数
@my_decorator
def say_hello():
    print("Hello!")

say_hello()
-------
这样执行后的结果say_hello = my_decorator(say_hello) 所以原来的say_hello还在，他是定义，而现在的同名函数变成了一个实例化的函数，而又由于这个高级函数，是以入参函数执行内部函数的，就是等价于了执行 wrapper()。 所以装饰器的意思就是为一个函数套一层框架，而这个框架的函数内容是固定的，不同函数用这个装饰器，就会附加上同一个内容，而自已的内容可变，进而能做到日志记录，缓存，权限校验的内容。

而进一步的装饰器也可以带上参数，用于给内部函数附加参数。
def repeat(num_times):
    def decorator(func):
        def wrapper(*args, **kwargs):
            for _ in range(num_times):
                func(*args, **kwargs)
        return wrapper
    return decorator

@repeat(3)
def greet(name):
    print(f"Hello {name}!")

greet("World")

其装饰的结果就是 greet = repeat('3')wrapper(greet()) 多一层嵌套了。
ex:
类装饰器，用于修改类的行为或是类的属性。
def class_decorator(cls):
    cls.class_attr = 'Added by decorator'
    return cls

@class_decorator
class MyClass:
    pass

print(MyClass.class_attr)  # 输出 'Added by decorator'

-----
装饰器定义的名称是外部函数的名字，因为闭包的存在，我们装饰器附加的函数会作为外部函数的参数，进而能被内部函数所调用，而且这样可以通过调用不同的装饰器函数 + 内部函数，实现不同的功能。
总结来说，装饰器的存在就是为了实现函数的入参，通过不同的入参，配合同样的内部函数实现，增加代码复用性

当然我们还会见到其他的装饰器，property装饰器，它可以把方法伪装成属性
class Temperature:
    def __init__(self, celsius):
        self._celsius = celsius

    @property
    def celsius(self):
        return self._celsius

    @celsius.setter
    def celsius(self, value):
        if value < -273.15:
            raise ValueError("温度不能低于绝对零度")
        self._celsius = value

    @property
    def fahrenheit(self):
        return self._celsius * 9/5 + 32


temp = Temperature(25)
print(temp.celsius)      # 25，看起来像属性访问
temp.celsius = 30        # 实际上调用了 setter
print(temp.fahrenheit)   # 86.0，只读属性

3、yield是用于生成器的一种关键字，和return类似，从函数返回相关内容。但是不同的是，yield返回的值可以在下次函数调用时在yield位置的后面继续执行，而不是退出函数。因此是因为有了yield才使得函数变为生成器，可以逐步迭代处理大数据或是流数据而不用一次性占用大量内存
--------
def simple_generator():
    yield 1
    yield 2
    yield 3

gen = simple_generator()

print(next(gen))  # 输出: 1
print(next(gen))  # 输出: 2
print(next(gen))  # 输出: 3
------------
在这个例子中simple_generator是一个生成器函数，每次调用next就会在yield处停止，返回一个值，下次调用后会从上一个停止位置继续执行。

生成器不仅可以向调用者返回值get，也可以从调用者接受post，这通过生成器的send()方法实现，x=yield，这就是接收的部分
send(10)传递参数后，其也会执行至yield的位置。

生成器是一个内存友好的迭代器，其一次只会生成一个值，在处理大型数据集时，不像列表一样占用大量内存空间，因为你可以等到遍历使用的时候，再把内容生成出来。比如我们可以定义
```python
my_gen = (x * x for x in range(10))
for num in my_gen:
    print(num)
```
这样每一次输出的时候，只会有一个元素的内存占用，但是却拥有完整的内容。

所以进一步的生成器遵循迭代器协议，可以使用for循环直接迭代生成器，就是一步步这样的执行，就像打断点一样的调试。
所以说迭代器是一个对象，实现了iter()返回迭代器对象本身和next()返回容器的下一个元素。

对于我们的项目来说我们使用了异步生成器 1是通过astream_events记录langgraph运行图中发生的事件，用于实现LLM中思考过程的内容，2是通过返回的特殊标记，决定将康复指南落库，3、通过Exception，可以接受错误信息返回前端。这样执行的时候只需要运行一次，而每次执行到特定结点的时候，我们都会配合Server-Sent Events (SSE) 协议，实现了零延迟的打字机流式推送效果。
async def ... yield 然后关于Langgraph的调用都使用Await，表示我们每一个协程都需要等待Langgraph有所回应才会执行下一步内容，然后如果有多个API请求过来，就都放入事件循环中。

这样最终呈现的效果就是，多个API请求过来，多个协程同步执行LLM的这些I/O密集型操作，挂起，而每一次运行结束后都会唤醒对应协程，然后走入下方的事件判断，通过设计好的输出，经由SSE传递给前端，然后 经由yield返回， 这样事件也是可以中断的， 比如请求停止后，那么yield也不会继续执行，进而停止在对应步骤上了。

4、GIL (全局解释器锁) 与并发编程
一、Python的线程池：
为了实现并发编程，自然需要构建多线程，讲不同的任务分配给不同的线程，那么线程池用于提高多线程应用程序的性能。它的设计和PostGreSQL，Redis，Kafka这些中间件的设计思路一致。
1、线程池的初始化，在程序启动时，线程池会预先创建一定数量的线程，这些线程被保存在内存中，不消化CPU资源，占用较小的内存空间，等待任务。（比如PostgreSQL，为了防止多次连接数据库，也会把连接预先存储，等到需要存库的时候直接用这些连接，KafKa一样的，他们都使用队列将各种任务分配给相关的连接。）
2、任务的提交，当新任务需要执行时，不用创建新进程了，而是提交到线程池来执行，通过任务队列管理，所有待执行的任务都被存储在任务队列中。
3、任务的分配和执行，线程池中的线程会不断轮询任务队列（因为执行的是线程，而不是任务，所以是线程去找任务）检查新任务，找到新任务，线程会从队列中取出任务并执行。这些任务执行是并发的，多个线程同时执行不同任务。执行完后的结果也会存储，通过Future对象获取。
4、线程的重用与回收，完成任务执行，线程会返回线程池并等待下一个任务的到来，避免频繁创建和销毁线程的开销。

所以对于I/O密集型任务，比如网络请求，LLM调用，文件读写等，这些任务的关键结点不在CPU的使用，而是外部资源，那么可以通过多线程来提高效率。此外需要注意：
线程池是一种并发的策略，他与异步是不同的，虽然两者可以混合使用，异步的定义是通过协调多个任务的执行，减少阻塞事件，就比如前面的yield执行LLM推理过程，我们异步执行事件图的各个结点，然后对于多个用户使用我们的Agent的时候，对于其中的请求任务和数据库任务，可以使用并发来提高效率。



二、Python中的多进程和多线程的应用场景以及优缺点：
多线程就是我们前面提到的，用于I/O密集型任务，提升程序的并发能力
优点是线程之间共享内存空间，通信成本低，创建和销毁进程的开销小，
缺点是：受限于Python的全局解释器锁GIL，不能充分利用多核CPU的优势，且线程之间共享全局变量，需要处理安全问题。
这里需要注意多核CPU的优势是用在处理计算密集型的任务，比如大矩阵乘法的操作，但是我们的项目是为了处理数据库和API请求耗时长导致的等待问题，他们本来就不是计算密集型，那这个缺点也不存在了。我的检索模块是I/O密集型（I/O-bound）任务。在请求 ChromaDB 或者发送网络 HTTP 请求时，Python 线程在等待响应的期间会主动释放 GIL。所以我们把检索中的向量查询和BM25关键字查询操作使用了多进程并发调用。

多进程，主要用于CPU的计算密集型任务，比如大规模数据处理，复杂计算和图像处理，每一个进程有独立的GIL，利用多核心并发执行任务。
优点：每个进程有独立的内存空间，不受GIL限制，可以充分利用多核CPU，进程之间独立，不会互相干扰。
缺点：进程间的通信复杂且开销较大，创建和销毁进程的开销较高。

Extra：GIL 全局解释器锁，其是一个用于解决多线程对Python对象进行同时操作而存在的一把全局锁，保证同一时刻只有一个线程执行Python字节码，虽然确保了线程安全，但也限制了多线程程序在CPU计算密集型任务的性能。
比如说我们有8个线程，由于GIL的存在，保护其共享空间内的参数的安全，底层CPU执行的时候，也是轮流执行线程，而不是实现真正的并发执行，所以对于计算密集型任务，这个操作是多余的，但是对于I/O密集型，这个操作就是可行的， 因为其他线程CPU执行后会遇到I/O中断，自行结束，也就释放了GIL，其他线程就会执行。


5、异步编程中的协程和线程：
对于I/O密集型任务，我们可以用AsyncIo做异步编程，其是单线程的，靠事件循环调度协程，没有线程切换开销
其核心语法为 async/await关键字，和协程（一个在线程中通过事件循环模拟出多个线程并发的效果）

1、使用async def定义的函数是一个协程，内部可以使用await关键字，使用async def定义的函数，调用之后的返回值也是一个协程对象，可以用于await和asyncio.run。语法层面上看，async def定义的函数是一个协程函数，但是运行时其类型仍是function，调用后返回的对象是一个协程对象

2、await + 协程：当我们对一个协程使用await的时候，当前函数【任务】中断执行，Python解释器开始执行协程的代码。
async def async_hello_world():
    now = time.time()
    await asyncio.sleep(1)
    print(time.time() - now) # 1.0013360977172852
    print("Hello, world!") # Hello, world!
    await asyncio.sleep(1)
    print(time.time() - now) # 2.0025689601898193

print(asyncio.sleep(1)) # <coroutine object sleep at 0x102f663b0>
coro = async_hello_world()
asyncio.run(coro)

------
对于基本的使用，await和直接调用函数是一个意思，
------
async def main():
    await asyncio.gather(async_hello_world(), async_hello_world(), async_hello_world())

now = time.time()
# run 3 async_hello_world() coroutine concurrently
asyncio.run(main())

---------
但是如果我们把这些协程都放入同一个协程函数，此时执行的结果，虽然只有单个线程被CPU操作，但是他们确实并发执行的，他借助事件循环用单个线程模拟多个线程并发。
---------

3、await + Task 在异步编程中，真正并发的对象是任务，我们上述的asyncio.gather（）就是把这些函数装载为一个任务列表，当我们对一个Task执行await的时候，event loop会开始调度当前可执行的全部任务，直到被await的Task结束。
其定义如下await asyncio.gather的意思是异步执行内部的任务，
await asyncio.gather(async_hello_world(), async_hello_world(), async_hello_world())
 <=====>
    task1 = asyncio.create_task(async_hello_world())
    task2 = asyncio.create_task(async_hello_world())
    task3 = asyncio.create_task(async_hello_world())
    await task1
    await task2
    await task3
其内部的处理流程是一、执行任务的装载，整个main()函数是第一个协程对象，他等待执行后续的任务。二、我们定义了三个任务，并将其装载后，任务列表中出现了hello_world的三个任务。Task 1 2 3，且他们都处于函数的第一行。三、eventloop执行各个任务，此时执行await task1，此时中断上一个任务Task0，【也就是进入了调用的函数（协程）】执行Task1的内容，当Task执行到await asyncio.sleep（1）的时候，Task1就会中断，转而执行Task2。循环完毕后，三个任务都执行完了一二行代码，进入sleep阶段，（这里注意，eventloop是一直在执行的，只是没有任务，所以他也是等价于sleep状态）四、再次执行，这里因为Task1最先sleep，所以他的任务是最早可以执行的事件，所以之后的操作，再重复一次【三】的内容。五、Task1最后执行完毕走到await Task2，中断Task0内容，先执行Task2，最后的完成顺序是 1 2 3 0.
------------------------
总结一下：异步编程的关键在于Await的使用，await的意思是 对于我们当前定义的这个协程，后续的这个操作很耗时，我们需要暂时放弃当前协程对于当前线程与CPU的占用，把当前任务挂起，这样任务队列就会继续循环，找到下一个可用的协程。
几个关键易错点：
1、Asyncio自始自终只使用一个线程，它的任务跳转是事件循环在这个线程中来回切换执行不同的协程函数，不是操作系统级别的线程切换，也不是多线程。
2、协程有着挂起和正常结束，中断结束等几种方式，（和操作系统挺像的）挂起的意思是，这个协程遇到了一个耗时任务，他还没有执行完毕，但是决定让出线程。而当其等待的Awaitable对象 -Future返回了“已完成”状态，那么Future返回的结果就会被事件循环检测，进而重新唤起协程。正常结束，当整个协程执行完毕，这个任务就进入结束状态，也可以从任务队列里面退出。中断结束当我们中断一个协程的时候 cancel调用，会引起Task对象向封装协程抛出CancelledError异常，等到下一次执行到这个Task的时候，抛出终止并取消该任务。
3、因为Await才是协程交出CPU的关键，如果异步编程中，一个协程一直不使用await会一直占用线程，如果这个运行的协程是死循环，也会直接卡死整个事件循环。
4、作用：异步编程的协程的作用，就是让等待的时间被复用，在 Agent 系统中，90% 时间都在等网络、等模型、等数据库。这些等待是系统级的，asyncio 在这些空档期切到其他任务，实现"一个线程处理成千上万个连接"。在 LangChain 调用大模型 API 时，网络传输是最耗时的部分，通过在网络请求处使用 await，我可以让程序在‘等网络数据包’的空闲时间里，去处理其他 Agent 的逻辑或用户的输入，这样这些任务的总耗时约等于其中耗时最长的协程。
5.Extra：如果我们还想在异步编程里面处理大量CPU计算，就需要结合多进程了，把当前任务分片，交给多个子进程，但是需要处理跨进程通信的问题了，因为协程只运行在单线程单进程上，它的CPU利用率是很低的，。

6、内存管理 与 垃圾回收：

内存管理的核心时是 引用计数 + 分代垃圾回收

每一个对象内部都有一个引用计数器，有人引用 + 1，引用断了 -1 降到0的时候释放
import sys

a = [1, 2, 3]
print(sys.getrefcount(a))  # 输出 3，a 本身 列表引用 + getrefcount 参数一次引用 + 一次临时引用

b = a
print(sys.getrefcount(a))  # 输出 4，多了 b 这个引用

del b
print(sys.getrefcount(a))  # 回到 3

但是他解决不了循环引用的问题， 比如说他们会出现在内部变量中
class Node:
    def __init__(self):
        self.parent = None

# 创建循环
node_a = Node()
node_b = Node()
node_a.parent = node_b  # A 引用 B
node_b.parent = node_a  # B 引用 A

# 即使删掉外部变量，内部依然互相持有
del node_a
del node_b

虽然我们已经删除掉了这两个引用，但是node_a内部的Parent属性指向node_b，这回导致node_b的计数+1，同理node_a计数也+1，此时我们发现，因为没有显式的删除内部引用，这下引用计数无法-1，也就无法释放，进而导致内存泄漏。
-------
引用计数的底层实现，Cpython的每个对象的C结构体头部都有一个ob_refcnt字段（引用计数器），
1、赋值b = a 此时a指向的对象ob_refcnt + 1 【请注意！！是被引用的对象计数加1 ，简单记忆，赋值号的右边计数+1】
2、函数传参，形参指向实参对象， 这个时候实参对象的引用计数+1，
3、放进容器 比如 lst.append(a) 这个时候a的引用计数器+1
4、删除，del语句，变量重新赋值，离开作用域，都会使得引用计数器 - 1.
只要变量的引用计数为0就会立即释放内存。但是因为这个计数器每次赋值与传参都会修改，在多线程下就需要加锁保护，因为线程执行顺序不固定，不是原子操作，如果释放的次数超过了引用次数，后面的执行全部错误，这就是GIL全局解释器锁的作用。
-------

分代垃圾回收，为了解决循环引用，Python引入了基于标记 - 清除算法的垃圾回收器
这个回收器不会关注计数信息，而是全局扫描
根对象（Roots）：从所有当前活动的变量（如全局变量、栈帧中的变量）出发。
标记（Mark）：顺着引用链，把所有能到达的对象标记为“存活”。【这个就是关键部分，因为手动的处理会漏掉引用】
清除（Sweep）：没有被标记的对象（即那些孤立的循环引用环），统统判定为垃圾，强制回收。
这里的意思是把遍历所有容器对象，每个对象的引用计数复制一份作为副本，然后把容器内部的引用从副本减去，如果计数归零说明就是我们前面提到的循环引用，可以回收。

另一个问题，为什么要分代？
为了性能。刚创建的对象大概率很快消亡，老对象大概率长期存活。
第 0 代（年轻代）：新对象。扫描频率最高。
第 1 代（中年代）：熬过一次 GC 的对象。
第 2 代（老年代）：存活很久的对象。扫描频率最低。
注意，这里的意思是，垃圾回收的扫描是一直在进行的，新对象创建后就是第0代，如果它的引用计数超过了700，说明他是一个需要长期保留的，就会晋升第一代，而在第一代中经过了10循环就会进入第二代。

此外，自带循环引用的对象，比如list套list，和某些内置类型，str/int/tuple等不可变对象，因为其有专门的优化或缓存机制，他们都不受分代垃圾回收处理。

总结：内存管理在Agent的应用场景：

在开发Agent系统时，内存管理从"知识点"变成了"生存技能"。你需要同时应对：
大模型权重：7B模型约14GB，13B模型约26GB

1、大模型加载的时候，我们需要先清理已有模型的引用信息，并手动执行gc.collect()回收，而在使用完毕后，也需要删除模型的引用。比如说我们加载的模型参数和Tokenizer。

pytorch相关内容：Autograd机制详解：
1、Pytorch的关键在于计算图，autograd就是一个动态计算图，他会在每次前向传播中构建一个新的计算图。
但是需要注意，我们计算的数据必须是tensor类型，有梯度grad才能进入计算图，而对于其中的乘法与加法计算等操作也会有相对应的节点。通过标准的Python控制流和运算，可以构建特定的计算图，它的自动求导（Autograd）机制通过记录张量（Tensor）的操作轨迹，生成一个有向无环图（DAG）。在反向传播（backward()）时，通过链式法则从根节点自动计算梯度，并累加到叶子节点的 .grad 属性中。”
2、autograd的三大核心组件：
一、梯度属性，1、requires_grad决定了当前张量会不会进入计算图，2、is_leaf是由用户的创建的原始信息，还是计算中间量。3、grad_fn定义了梯度的挂载在计算图的哪一个结点上。
二、梯度计算与链式法则，梯度的计算是dy/dx，链式法则则是求导计算的链式计算法则 fx(f(y))' = fx'(fy) * fy'
三、梯度累积机制：每一个epoch之后，我们的标准操作都是1、计算梯度，2、反向传播，3、清空梯度。当然这里也有特例，比如GAN的训练需要多次反向传播，或是计算高阶导数的时候，我们在反向传播的时候使用retain_graph保留计算图，或是不清空梯度，让梯度累积。
四、detach()和no_grad()的区别，detach常用于从计算图中分离数据，返回一个不需要计算梯度的新的Tensor，比如我们需要借助中间变量放入其他部分使用，而no_Grad用于验证部分，直接禁用这一阶段的梯度计算，即在前向传播的时候不需要保存中间的激活值也不用构建计算图，进而降低显存占用并提升推理速度。
3、数据处理部分，比如模型训练时候GPU利用率上不去，经常等待CPU，会这么排查和优化，
出现CPU等待，就是数据加载缓慢，解决方案可以是1、在Dataloader中调大num_workers，开启多进程的数据处理，2、我们采用的方法，数据预处理的部分直接离线处理，并转为Tensor格式，使用的时候直接提取。

大模型相关的内容：
Transformers是什么，它的整体结构是什么样子的：
答：
Transformer 是 2017 年 Google 提出的一种基于纯注意力机制的架构，核心思想是用 Self-Attention 替代 RNN 的递归计算，这样序列中的任意两个位置可以直接交互，而且计算可以并行化。

原始结构是 Encoder-Decoder 形式。Encoder 端每层是 Multi-Head Self-Attention 加上 FFN，中间有残差连接和 LayerNorm。Decoder 端在此基础上多了一个 Cross-Attention 来关注 Encoder 的输出，同时 Self-Attention 加了 Causal Mask 掩码矩阵，保证自回归生成时只能看到前面的 token。

不过现在主流大模型，比如 GPT、LLaMA、Qwen 这些，基本都是 Decoder-Only 架构，把 Encoder 去掉了，只保留带 Causal Mask 的 Decoder 部分。Encoder-Decoder 架构在翻译、摘要等 seq2seq 任务里还有用武之地，比如 T5 和 BART。

真正让 Transformer 成为基础设施的原因，一个是并行训练效率远超 RNN，另一个是它的 Scaling 特性非常好，模型越大、数据越多，性能提升越明显。

问：
Decoder-Only 为什么比 Encoder-Decoder 更适合大模型？（→ 涉及统一建模、Scaling 效率）
答：
最核心的原因是训练信号密度。Decoder-Only 的 next token prediction 目标下，每个 token 都产生 loss，n 个 token 有 n-1 个训练信号。BERT 的 MLM 只 mask 15%，信号密度差了将近 7 倍。同样的数据和算力下，Decoder-Only 学到的东西更多。
loss = cross_entropy(model(input[:-1]), input[1:]) 它的损失是一个自回归函数，就是通过前文的序列[BOS] + 输入序列 + [SEP] + 输出序列计算得到下一个token的预测。

第二是 Scaling Law。GPT-3 和 Chinchilla 等大规模实验反复验证了 Decoder-Only 在 scaling 时性能提升最平滑、最可预测。对于几百万美金级别的训练决策来说，"可预测"这件事极其重要。
这也是和架构有关，因为传统的编解码器架构中，Encoder需要先把输入压缩为固定长度的上下文向量，而Decoder又要在算自注意力的基础上再额外计算encoder结果的交叉注意力，推理效率下降了

第三是任务统一性。所有任务都可以转成 prompt 加 completion 【前文 + 续写】的形式，不需要区分 encoder 和 decoder 的输入。工程上也更简单，推理时 KV Cache 的使用非常直接。

当然 Encoder-Decoder 在翻译、语音这些特定场景还有优势。Decoder-Only 也不是不能做理解任务——通过充分训练和 instruction tuning，causal 约束下的理解能力也很强。

总结就是，作为通用基座模型，Decoder-Only 在效率【计算复杂度低了，少了关于encoder的计算和Decoder对Encoder输出的交叉注意力计算】、scaling【模型参数大，性能的提升还是稳定的】、统一性【所有任务都能通过Prompt + completion的方式，借助自回归任务完成训练，在训练信号密度上也较高】上全面领先，这就是为什么所有头部模型都用这个架构。

问：残差连接和 LayerNorm 的位置有 Pre-Norm 和 Post-Norm 之分，有什么影响？
Post-Norm指的是，先残差后归一化  x = LayerNorm(x + Sublayer(x))
缺点：残差分支被包裹在了归一化层里面，随着模型加深，主干网络的值会不断被归一化，这就导致深层网络的梯度容易消失或是爆炸，在训练时，必须依赖非常谨慎的 Learning Rate Warmup（学习率预热）才能勉强收敛。
∂loss/∂x = ∂loss/∂LN × ∂LN/∂(x+attention) × (1 + ∂attention/∂x) 梯度必须经过LayerNorm，而LN的梯度可能不稳定

相应的PreNorm 是先归一化后再残差计算， x = x + subLayer(LayerNorm(x))
优势: 存在直连路径，梯度可以直接通过残差连接回流 就是直接去除掉后续的LN部分也可以，
数学: ∂loss/∂x = ∂loss/∂残差 × 1 + ∂loss/∂attention × ∂attention/∂LN × ∂LN/∂x
这样残差的主干路径可以以 x = x + .. +... 这样干净的线性恒等映射，无论堆叠多少模型进行残差计算，梯度都可以无损传导回来。
Pre-Norm 极大地提升了超大参数量模型训练的稳定性，去除了对 Warmup 的强依赖。现代模型甚至在 Pre-Norm 的基础上，为了追求极致的计算速度，进一步将 LayerNorm 替换为了没有均值平移的 RMSNorm。
就说白了，一个是嵌套逻辑，每一次计算都会把归一化嵌套，而pre-Norm则是累加的形式，那么基于链式法则，加法肯定更加简单。

问：
Transformer 的计算复杂度是多少？长序列场景下怎么处理？
答：计算复杂度的关键部分在于自注意力部分，我们设序列长度为N，隐藏层维度为d，此时计算的大头就集中在QK矩阵的计算上，时间复杂度为O（N^2 * d), 空间复杂度为O（N^2），所以在长序列下，显存和计算量都是需要考虑的部分。
1. QKV投影: O(n × d × 3d) = O(3nd²)
2. QK^T计算: O(n × d × n) = O(n²d)  ← 主要瓶颈！
3. Softmax: O(n²)
4. 加权V: O(n² × d) = O(n²d)
5. 输出投影: O(n × d × d) = O(nd²) 当然这里的d^2还有常数，取决于投影的转换维度倍数。

总复杂度: O(n²d + nd²)
当 n > d 时，O(n²d) 主导

对于优化方案来说主要有以下4种：
1、底层算子优化，通过分块计算和重计算，避免将巨大的O(N^2)矩阵写回GPU的显存中计算，只在SRAM内存中计算，将空间复杂度从N^2变为N，大幅提升计算速度。
2、KV cache显存优化（MQA/GQA/PagedAttention），KV cache虽然是在时间复杂度上减少了，但是它的空间复杂度无法解决，所以利用MQA/GQA通过多个头共享参数的设计，减少Key和Value头的数量，部署上，使用分块策略，就是每次只计算一部分的QK矩阵并存储。
3、架构降维，直接在计算层面减少，比如说稀疏注意力（每一个Token只关注部分其他Token，比如滑动窗口），线性注意力（借助特征映射，借助这个理论【高维度的映射等价于使用激活函数对特征进行变换】，因此QK的计算从一个Token需要与长度为N的所有K计算变成了只需要计算一次，这样n*d * n*d -> n*d * 1*d这个意思，这样复杂度少了一个数量级N）
4、位置编码扩展，RoPE的外推，正如我们前面对于位置编码的部分，绝对编码对于每一个位置都是一个固定的值，而相对编码的其他方式都需要在注意力计算的时候赋予，而旋转编码只需要添加特定的旋转矩阵即可。这是一方面。此外，RoPE的旋转矩阵可以通过插值变换，扩展使用范围，比如训练只有4K，但是推理时却能执行32K的长度，模型无需从头训练就能理解更长的相对位置。



Transformers的自注意力机制：
Scaled Dot-Product Attention ：通过计算矩阵Q和K的值并softmax以及除以sqrt(dk)维度，【因为你这样想，矩阵QK里面的值都会因为维度的长度而多计算dk次，它的值的方差是很大的。】
QKV矩阵代表了同一输入的不同视角投影，
Query（Q）："我要什么"​ - 当前 token 的信息需求
Key（K）："我有什么"​ - 每个 token 的标识特征
Value（V）："我提供什么"​ - 每个 token 的实际内容
这里的关键在于K和V的拆分，K表示这个向量的标识信息，V标识具体的语义信息，Q代表了需求信息，【这里我们的例子我们的Q和KV不是同一个Token上的内容，因为需求是我们当前这个词的信息，而他可以寻找的信息可以在前文的所有内容中的K和V】
这样QK的计算是得到了需求与标识的匹配度，而他们匹配度加权后提取各个V的值，就是得到了实际内容
在句子"我喜欢苹果"中：
当处理"喜欢"时，它的 Q 寻找情感对象
"苹果"的 K 表明它是名词/实体
"苹果"的 V 包含"水果"的语义信息
注意力权重高 → 提取"苹果"的 V → 理解"喜欢苹果"  所以这得到了一个效果，他学习到的是情感词和名词的对应，后续我们换了不同的名词，他还是知道要这么找名词的效果。
Q、K、V 三个矩阵各有明确的分工。Q 表示"当前 token 在找什么信息"，K 表示"每个 token 能提供什么样的匹配信号"，V 表示"匹配上以后实际要传递的内容"。

如果只用一个矩阵，等于不做任何投影直接在原始空间算相似度，模型没有能力学习"什么算相似"。加了 Q 和 K 两个投影后，WQW K^T本质上定义了一个可学习的相似度度量，不同的 head 可以学到完全不同的匹配模式。而且 Q 和 K 分开保证了注意力是非对称的——"A 关注 B"和"B 关注 A"可以有不同的程度，这对建模语言结构很重要。

V 的独立投影让"匹配"和"传递"解耦——怎么算相似度是一回事，匹配上以后传什么信息是另一回事。两者在不同子空间独立学习，表达能力更强。

虽然后来 MQA 和 GQA 让多个 head 共享 K 和 V，但那是效率和表达能力之间的权衡，不是说 K 和 V 不重要。

为什么 MQA 选择共享 K 和 V 而不是共享 Q？这说明了什么？
答：第一，从注意力机制的本质看，Q 代表了多样化的查询视角，每个头应该从不同角度理解输入，这是多头注意力的核心价值。而 K 作为匹配标识、V 作为内容信息，更具有客观性和共享性。
第二，从工程瓶颈看，在自回归推理时，KV Cache 的内存访问是主要瓶颈。MHA 需要为每个头存储独立的 K/V 缓存，导致显存占用和读取带宽都乘以头数 h。共享 K/V 后，KV Cache 从 h 份减少到 1 份，显存占用降低到原来的 1/h，这是最直接的优化收益。
第三，从优化效果看，如果共享 Q 而保留独立的 K/V，KV Cache 的瓶颈依然存在，却牺牲了模型的核心表达能力，得不偿失。实验证明，MQA 在性能下降很小（1-2%）的情况下，能带来 30-40% 的推理速度提升。

其实可以这样考虑，信息的内容毕竟是固定的，但是如何通过查询询问这些内容的不同方面是重要的，（我们是在MQA，也就是多头设计下的处理，所以共享KV不是把KV都扔掉了，只是我们不打算对他拆分出多份信息而已。）

Extra：如果把Q和K投影到相同维度空间会怎样：
答：使用同一个投影矩阵的话，QK计算的结果将变成对称矩阵，虽然还能保证基本的Token之间的匹配模式，但是对于A关注B的程度将会与B关注A的程度完全一致。 比如说，代词"it"应该是强烈关注他指代的名词，但是名词是不一定需要同等程度关注代词'it'的，所以说，不同的QK，所得到的非对称注意力分数，可以适配建模真实语言结构

如果把K和V投影到相同空间会怎样：
答：这会导致匹配的表示和最终传递的信息是同一个东西。这会损失灵活性——有时候我们想"根据语法结构匹配"，但传递的是"语义内容"。把 K 和 V 分开，模型就能独立学习"怎么匹配"和"传递什么"。就像我们前面说的例子，或者那情绪识别距离吧，我们要学习到的是方面词 - 情绪词之间的对应关系，这是K做到的内容，V做到的内容是，只要我们有了这个关系，情绪词是什么，方面词是什么我们都能识别到，如果他和K是一样的，他识别到的内容就会与训练数据完全一致，而没有普适性。

问：多个投影和单个投影的对比：
可以这样总结：单矩阵方案让"查询"、"索引"、"内容"三者耦合，所有功能在同一个空间完成，表达能力受限。三矩阵方案让三者解耦到各自的子空间，模型可以分别学习"怎么提问"、"怎么被匹配"、"传递什么信息"。这种解耦在工程上增加的参数量很小（只是三个线性投影），但带来的表达能力提升很大
我们的论文也做了同样的操作，通过将同样的Encoder融合信息，解耦成具体的语义信息，提升了表达能力。

总结来看：
Self-Attention 的计算流程分几步：首先对输入做三次线性投影得到 Q、K、V；然后 Q 和 K 做矩阵乘法得到 n×n 的注意力分数矩阵；接着除以根号 d_k 做缩放；如果是 Decoder 架构需要加 causal mask；然后过 softmax 归一化成权重分布；最后用这个权重对 V 做加权求和，得到输出。

关于为什么要除以根号 d_k，核心原因是点积的方差会随维度线性增长。假设 Q 和 K 的元素是标准分布的，维度 d_k 的点积方差就是 d_k。如果 d_k 是 128，有些点积值就会很大。这些大值进 softmax 后会导致分布极度尖锐，接近 one-hot，一方面梯度消失训练不动，另一方面注意力过度集中在单个 token 上。除以根号 d_k 就是把方差拉回到 1 附近，让 softmax 在一个比较好的工作区间里。

直觉上可以理解为一种温度调节。不缩放等于用了很低的温度，分布太尖锐；缩放后温度合理，分布平滑。

位置编码的数学魔法：
为什么需要位置编码，对于我们这个论文就是最好的解释，因为一个句子的内容他是有长度的，内容相近的一定是相关度最高的，比如我喜欢香蕉，讨厌苹果，喜欢的动词已经学习到了适配名词，但是这里有两个名词， 哪一个是更好的，用上相对位置编码我们就知道一定是较近的这个是比较好的，而绝对位置编码只知道两者之间存在差异。

位置编码一般使用两种：一种是相对位置编码，另一种是固定的绝对位置编码。他们一般都采用正弦和余弦函数，为一个位置的，奇数和偶数次项的维度，赋予对应的数值，比如 0 2 4.基于sin编码，1 3 5基于cos编码，这样他们的数值就是一个可线性表示的值。但是线性关系会在经过多层Attention计算后被稀释。对于绝对位置编码还有把编码信息弄成可学习的向量，

所以在绝对位置编码中，由于位置的不同，编码也有不同的模式，帮助区分序列中不同位置的元素。其优势是简单且有良好可解释性，缺点是处理变长序列，（因为每次都要重新计算），长距离依赖时，无法充分表达复杂的位置信息，（只知道不一样，但是不知道是因为距离而产生的不一样。）比如我们打乱输入Token的顺序，QKV的行顺序跟着变，但计算结果不变，对于模型来说，他只知道这个序列有这些Token，但是不知道这个Token的顺序已经被改变了。

所以相对位置编码，不是直接为位置分配唯一编码，就是不想绝对编码一样在一开始就加上，而是在后续【注意力计算】的时候把位置编码加入，比如常见的直接将两个Token距离作为位置向量，他的意思就是用(i-j) * sim ,用相对位置 加上之前对维度奇偶次项的操作。也有ALiBi方法，直接在计算的相似度分数上减去一个与距离成正比的线性惩罚S[i,j]=qi*kj - m*| i - j |m是不同Head的斜率

还有一种是RoPE，即每一个token独立做旋转，但是点积结果只依赖距离差。因为旋转矩阵的正交性保证了两个Token做内积时，结果只依赖距离差，旋转编码只是对不同位置赋予了不同的旋转角度作为位置信息编码，这样两个不同位置之间的旋转角度会产生差异，但是计算的结果又不受影响。且实现上可以借助KV cache快速查表得到。简单来说就是旋转矩阵不同，体现了位置关系，而计算结果的不变保障了QKV注意力的计算结果。

而且注意到一个关键点 位置编码加在 embedding 上（加法）和加在 attention score 上（bias）有什么本质区别？
对于加在embedding上的，他是直接附加在了内容上，这会改变Token的信息，耦合两者信息在一个embedding里面。
而用在注意力分数上，则是不改变embedding的值，而是作为一个独立调节因子明确告诉模型不同位置之间应该如何关注，将位置和内容解耦，更高效的学习位置关系。
RoPE很特别，它既是只在计算注意力分数前的附加信息，但是又只改变了注意力分数的计算。

Extra：
如果一个模型训练时只见过 4K 长度的序列，直接推理 32K 会发生什么？
对于绝对位置编码，它的信息附加借助公式实现，可以直接推理，而可学习向量表示，由于只知道训练部分的长度，无法推理变长。
对于相对位置编码 bias 和 ALiBi都是只与距离有关，可以推理且实现简单，而RoPE，虽然也是之学习到的4K长度的旋转矩阵，但是可以借助NTK-aware scaling扩展上下文窗口。

总结来说：
位置编码的必要性来自一个核心事实：Self-Attention 是 permutation invariant（位置无关） 的，打乱输入顺序对 attention 计算没有影响。但语言是有序的，"我吃饭"和"饭吃我"意思完全不同，所以必须注入位置信息。

绝对位置编码给每个位置一个固定表示，典型的有 sinusoidal 和 learned。它的问题是编码的是"我在第几个位置"，长度泛化差，而且同一段文本放在不同位置会有不同的表示，不太合理。

相对位置编码关注的是 token 之间的距离。T5 用的是在 attention score 上加 bias，ALiBi 用线性惩罚，RoPE 用旋转。其中 RoPE 是目前最主流的，它用绝对编码的形式实现了相对编码的效果——每个 token 独立做旋转，但点积结果只依赖距离差。而且它兼容 KV Cache，配合位置插值方法还能做长度外推。

现在主流大模型基本都用 RoPE。ALiBi 也有一些模型在用，但整体上 RoPE 已经是事实标准了。

RoPE相关内容：

RoPE，Rotary Position Embedding，核心思想是把位置信息编码成对 Q 和 K 向量的旋转操作。具体来说，它把向量的每两个维度看成一个二维平面，根据 token 的位置乘以一个旋转角度。设计的精妙之处在于，旋转矩阵的正交性保证了两个 token 做内积时，结果只依赖于它们的位置差 m-n。所以形式上是绝对位置编码——每个位置独立旋转，效果上等价于相对位置编码。
qT*K = (R(mθ)q)T * (R(nθ)k) = qT * R(( m - n)θ)* K , 可以发现内积结果只依赖 (m - n)，对于更高维的内容，每一个使用不同的基础频率，就是绝对位置编码的 θ = 10000 ^ (-2*i/dk)，低维的子空间频率高（变化快，捕捉短距离关系），高维的子空间频率低（变化慢，捕捉长距离关系）。而且也两两分组，计算各个子空间的结果。
它适合大模型有几个重要原因。实现非常轻量，就是逐元素的乘加，不加参数不改结构。
常见的实现是把向量按奇偶维度拆分，然后用 sin 和 cos 做逐元素乘加，计算开销非常小，几乎和不加位置编码一样。具体来说，实现时通常预计算好每个位置每个维度的 cos 和 sin 值，存成一个表，推理时查表直接乘即可。
天然兼容 KV Cache，因为每个位置的旋转是独立的，只会于自已的位置有关。长度外推的基础好，配合 NTK-aware scaling 或 YaRN 这类频率调整方案可以扩展上下文窗口。
一共有三种常见方案：
Position Interpolation（线性插值）：把超出训练长度的位置线性压缩回训练范围内。比如训练长度 4K，推理 8K 时，把位置索引都除以 2。简单但不精细。
NTK-aware scaling：不是线性压缩所有频率，而是调整 base 频率θ（把 10000 增大），让高频分量保持不变（近距离关系不受影响），低频分量被压缩（远距离关系被外推）。
含义：高频外推指的是位置编码中的高频项在处理超出训练数据范围的位置时，仍能保持其快速变化的特性，从而捕捉到细粒度的位置信息。
原因：高频项变化迅速，能够捕捉到细微的位置变化。在超出训练范围（外推）时，高频项仍然需要提供细粒度的位置信息，以确保模型能够准确地理解和处理新的位置。
低频内插：
含义：低频内插指的是位置编码中的低频项在处理训练数据范围内的位置时，通过频率的调整，使其能够平滑过渡，从而捕捉到较大范围的位置变化。
YaRN：进一步精细化，对不同频率分量做不同程度的缩放，加上温度因子调节 attention 的集中程度，是目前长度外推效果最好的方案之一。

不过要注意，RoPE 本身不等于能做长上下文，直接外推到训练长度的几倍以上效果就会退化。实际操作中还需要位置插值方案加上少量长序列数据的继续预训练。另外 RoPE 只作用于 Q 和 K，V 不旋转，因为 V 参与的是加权求和而不是相似度计算。

MHA、MQA、GQA、MLA这些多头注意力机制有什么区别，
答：
MHA、MQA、GQA 的区别核心在 KV 的共享程度。MHA 是标准方案，每个 head 有独立的 Q、K、V，表达能力最强但 KV Cache 最大。MQA 走极端，所有 head 共享一套 K 和 V，KV Cache 压缩了 h 倍，推理很快，但精度可能有损。GQA 是折中，把 head 分成若干组，组内共享 KV。

实际效果上，GQA 质量非常接近 MHA，效率接近 MQA，所以现在主流大模型基本都用 GQA。LLaMA 2/3、Mistral、Qwen 都是这个方案。

从工程角度看，选择 MHA 还是 GQA 最直接的考量就是 KV Cache 大小。32 个 head 的模型，GQA 8 组可以把 KV Cache 压缩 4 倍，在长上下文场景下这个差距决定了能不能实际部署。一般来说分组数多为4和8，与头数和GPU数量相除无余数。

另外可以提一下 DeepSeek 的 MLA，它走了一条和 GQA 完全不同的路——不是共享 KV，而是把 KV 联合压缩到低维 latent 空间，推理的时候只缓存Latent向量，使用的时候再解压，压缩比更高，理论上表达能力也不损失，但实现更复杂。

Extra：如果想把已经使用MHA训练好的模型，转为使用GQA实现，就是需要把KV cache直接压缩。
需要先使用部分训练集收集相应的KV cache，基于余弦相似度，计算最优的正交矩阵，对QKV的投影矩阵运用正交变换，使得同一组内不同的Attention Head在特征空间中更加接近。基于此实现MQA的分组，然后通过剪枝训练，逐步将原始的 KV Head 转移到新的 KV Head 上。当然这些方法实际使用还是有缺陷，最好是一开始就用GQA。

LayerNorm 和 RMSNorm 有什么区别？为什么现代大模型偏好 RMSNorm？
简单来说：层归一化 和 只均值平方的层归一化
LayerNorm 做两步：减均值（centering）+ 除标准差（scaling），当然如果拆开来看的话，他对每个Token的隐藏层向量做了两次操作，计算他在每一层中的均值以及方差（求差的平方和再除以维度）然后还要做两次逐元素的操作，每一个Token再减去均值除去标准差，加上缩放以及偏移项，涉及了多步计算
而 RMSNorm 省掉了减均值那一步，只用均方根（RMS）做 scaling。意思是每一个Token不用减去均值，直接计算均值平方根用于缩放，同时去掉了偏移参数β实验表明减均值对性能贡献很小，但计算开销不可忽略，所以大模型普遍用 RMSNorm 来降低归一化的计算成本，同时效果基本不损失。因为归一化在每一个比如计算注意力后，前馈神经网络后，都会有归一化层。积少成多。

这些操作的原因都是因为大模型参数的增加，原先可能不太耗时的任务都已经变成累赘了，所以现在的方案是尽可能在不影响性能的情况下，减少不必要的操作。

Extra：如果完全去掉 Norm 层，模型会发生什么？Norm 在训练中到底起什么作用？
这会使得模型完全无法训练，因为模型的每次操作后的方差都会越来越大，这会导致下一层的输入分布剧烈变化，需要不断调整以适应这些分布，导致收敛缓慢。
其作用有4点，四大核心作用：1) 稳定器：保持激活值分布稳定；2) 梯度调节器：平滑梯度流动，允许更大学习率；3) 加速器：减少内部协变量偏移，加速收敛；4) 正则器：引入轻微噪声，提高泛化能力。LayerNorm特别适合NLP任务，因为它对每个token独立标准化，不受序列长度影响。
为什么常用LayerNorm而非BatchNorm。1、可以处理序列长度的变化的情况，2、不会依赖batch的统计数据的内容，3、更加适合自注意力

DeepNorm如何结合Post-Norm和Pre-Norm两者优点？
答：DeepNorm通过放大残差连接来结合优点。具体：1) 采用Pre-Norm架构，获得梯度直连路径；2) 引入缩放因子α>1放大输入，β<1缩小子层输出；3) 数学上保证输出范数有界，能训练极深网络（如200层）。在GLM-130B中验证有效。

问：
FFN 在 Transformer 中起什么作用？SwiGLU 相比 ReLU 有什么优势？

FFN 在 Transformer 里的角色，可以这样理解：Attention 负责 token 之间的信息聚合，FFN 负责对每个 token 做独立的非线性变换和知识检索。FFN 的参数量通常占整个模型的三分之二，可以认为它是模型存储知识的主要场所。有研究表明，直接编辑 FFN 权重就能修改模型存储的事实知识，这也侧面印证了 FFN 作为"知识存储体"的角色。

激活函数的演进线是 ReLU → GELU → SwiGLU。SwiGLU 有两个关键改进：一是用 Swish 替代 ReLU，更平滑没有死神经元问题；二是引入了 GLU 门控机制，多了一个投影矩阵做 gate，通过逐元素相乘让模型学会在特征维度上做选择性激活。PaLM 和 LLaMA 的实验都表明 SwiGLU 在同等参数量下效果一致更好。

工程上有个细节：因为 SwiGLU 多了一个投影矩阵，为保持参数量不变，隐层维度从 4d 调到了 8d/3。所以看 LLaMA 的代码会发现 FFN 中间层维度不是整 4 倍——比如 d=4096 对应的隐层是 11008，就是 8d/3 取整到 128 的倍数。
